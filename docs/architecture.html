1. High-Level Overview

FinanceStreamAI is a Kotlin/Jetpack-Compose Android trading-research app backed by a Python FastAPI service deployed on Render. The backend evaluates stocks across technical, fundamental, and momentum signals and surfaces matches for four option strategies: Cash-Secured Puts (CSP), Diagonal Call Spreads, Vertical Bull-Call LEAPS, and Long LEAPS. All persistent state (positions, watchlists, recommendations, learning stats) lives in Cloud Firestore. A weekly APScheduler job re-evaluates open recommendations and aggregates win-rate statistics by signal and verdict — a self-improving feedback loop. The Android client additionally cross-validates verdicts against up-to-five external LLMs (Claude, Gemini, ChatGPT, Perplexity, Grok) before display.

Client Highlights

  • Single-activity Jetpack Compose (Material 3)
  • Retrofit 2 + OkHttp 4 (X-User-Id header)
  • WorkManager: Daily scan + hourly portfolio-flip + scan workers
  • Google Sign-In via androidx.credentials
  • EncryptedSharedPreferences (AES256-GCM) for AI keys

Server Highlights

  • FastAPI + Uvicorn worker via Gunicorn (1 worker, 600s timeout)
  • Cloud Firestore for all state (no SQL DB at runtime)
  • Market data: yfinance + Tradier (with curl_cffi Chrome impersonation)
  • APScheduler: weekly evaluator (Mon 14:00 UTC) + startup-catchup
  • Async scan jobs (in-memory job table) + sync scan

3.4 AI Cross-Validator

AiCrossValidator.kt issues parallel requests (via coroutineScope { async }) to up to five LLMs whose API keys the user supplies. Each engine receives a structured prompt containing the ticker, price, current verdict, bullish/bearish signals, support/resistance levels, sector, and chosen strategies; each returns a verdict (BUY/SELL/HOLD/AVOID), confidence, and reasoning. The validator aggregates votes into a consensus (STRONG BUY / BUY / HOLD / AVOID / MIXED) and an agreement percentage. Results are cached in-memory for 15 minutes per ticker. API keys are stored in EncryptedSharedPreferences using AES256-GCM for both keys and values.

3.5 Authentication Flow

  1. User taps Sign In with Google on GoogleSignInScreen.
  2. CredentialManager.getCredential() launches the Google credential picker.
  3. Returned GoogleIdTokenCredential is parsed by GoogleAuthManager.handleSignInResult().
  4. User identity (id, name, email, photo) saved in GoogleAuthPrefs; UserSession.userId populated.
  5. An OkHttp Interceptor attaches X-User-Id: <uid> to every backend request.

4.3 JPFinanceEngine

The JPFinanceEngine class is the analytical core. Key methods:

  • prefetch_market_data(tickers) — bulk yfinance download (2y daily OHLCV) in parallel; warm fundamentals cache.
  • process_ticker(ticker) — orchestrates: technicals → fundamentals → IV rank → signals → verdict → strategy builders.
  • _get_csp_logic / _get_diagonal_logic / _get_vertical_logic / _get_long_leaps_logic — strategy match producers.
  • get_bt_success(history, strike_ratio, dte) — historical win-rate backtest.
  • _build_fundamental_signals(), _build_technical_signals() — signal lists feeding the verdict.

After each scan job, _history_cache.clear() and _fundamentals_cache.clear() run to keep the long-lived dyno’s memory bounded.

5.3 Startup Sequence

  1. Render boots dyno → Gunicorn launches 1 Uvicorn worker.
  2. FastAPI @app.on_event("startup") initialises Firestore client.
  3. SchedulerManager registers the weekly cron job.
  4. _check_eval_on_startup() runs in a background thread; if meta/eval_status.last_eval_at is > 6 days old, the evaluator runs immediately. This makes the weekly job resilient to Render’s ~24h dyno restarts.

    AND _has_quality_kicker
    AND _has_trend_confirm):
    verdict = "STRONG BUY"

    verdict = "BUY"

8.4 Early-Reject Gate

A ticker must satisfy at least one of the following or it is short-circuited (no strategy evaluation runs):

  • discount ≥ 8%, OR
  • RSI < 48 OR price ≤ lower Bollinger band, OR
  • price > SMA200, OR
  • confirmed/pending breakout, OR
  • ticker is in the curated large_cap_tickers list.

9. Option Strategies — Detailed Summary

9.1 Cash-Secured Put (CSP) — _get_csp_logic

Purpose: Sell downside protection and collect premium; willing to take assignment at strike.

  • Strike: 70–98% of current price
  • Expiry: ~60 DTE target (earliest available)
  • Delta: ≈ -0.25 (delta-neutral to slightly bearish)
  • Backtest (strike/price, 60 DTE): ≥ 80–85%
  • Monthly ROC: ≥ 2.0% strict; ≥ 4.0% large-caps; ≥ 5.0% crypto-related
  • Hurdle gate (must satisfy ≥ 2):
    • discount ≥ 8%
    • RSI < 48 OR price ≤ lower Bollinger
    • price > SMA200
    • confirmed/pending breakout with volume surge ≥ 1.2×
  • Liquidity: reasonable bid/ask spread
  • Exits: assignment at strike (acceptable); stop-loss at strike − 1×ATR; roll if breached or DTE < 60 and profitable

9.2 Diagonal Call Spread / PMCC — _get_diagonal_logic

Purpose: Synthetic covered call — collect premium on the short call while owning a cheap deep-ITM long call.

  • Long leg: ≥ 240 DTE · strike < 95% of price · delta 0.62–0.85 · sufficient intrinsic value
  • Short leg: 40–75 DTE · strike > current price · long_premium / short_premium ≥ 17%
  • Net debit = long cost − short credit; max profit ≈ intrinsic spread
  • Hurdles: if BT ≥ 85% need ≥ 1; if BT ≥ 70% need ≥ 2 of:
    • discount ≥ 5%
    • RSI 30–72 (neutral/rising)
    • price > SMA20
    • breakout pending/confirmed OR momentum trending
  • Exits: short strike breached → close/roll short leg; support broken → exit both; roll short at 21 DTE

9.3 Vertical Bull-Call LEAPS — _get_vertical_logic

Purpose: Defined-risk directional bet on large caps.

  • Universe restriction: only tickers in the curated large_cap_tickers list (e.g., NVDA, MSFT, META, AMZN, TSM …)
  • Expiries: 2027–2028 LEAPS only
  • Long leg: strike < 98% of price · delta 0.55–0.75
  • Short leg: strike > 102% of price · delta 0.18–0.35; (long_strike + net_debit) ≤ price × 1.15
  • Backtest (breakeven = long_strike + net_debit): ≥ 80%
  • Profit: strike_width − net_debit; Loss: capped at net_debit × 100 per contract
  • Exits: stock reaches short strike (max profit); support broken (stop-loss); roll at 30–45 DTE

9.4 Long LEAPS — _get_long_leaps_logic (Jason L. Capital cheat-sheet)

Purpose: Cheap leveraged long-only directional exposure (synthetic stock replacement).

Setup must satisfy ONE of:

  1. Classic deep discount: discount ≥ 10% AND price > SMA200
  2. Confirmed breakout: breakout_confirmed OR (breakout_pending AND volume_surge ≥ 1.2×) AND price > SMA200
  3. Strong trend: price > SMA200 AND price > SMA50 AND momentum_20d ≥ 5%

Strike selection

  • Option A — ITM Deep: strike < price; intrinsic buffer ≥ 60% of premium; delta 0.75–0.85
  • Option B — ~10% OTM: strike 0–12% above price; delta 0.45–0.60 (cheaper, still directional)

Other gates

  • Expiry: 600+ DTE preferred (360+ minimum; < 180 DTE rejected)
  • Multi-timeframe RSI: daily < 40 + weekly curling up = premium entry; weekly ≥ 75 = overbought warning
  • IV Rank: < 45% (avoid paying for vol crush)
  • Backtest (378 DTE): Classic ≥ 75%, Breakout ≥ 62%, Trend ≥ 68%
  • Fundamental quality: at least one of — analyst buy, revenue growth > 20%, earnings growth > 25%, gross margin > 60%, operating margin > 25%

Exits

  • First target = resistance level (52w high or technical resistance)
  • Stop-loss = support level OR price − 2×ATR
  • Roll if expiry < 60 DTE and profitable
  • Close if support breaks or thesis invalidated

10. AI Feedback & Learning Loop (ai_feedback.py)

10.1 RecommendationTracker

  • capture_scan_results(results) — de-dups by (ticker, strategy, strike, date) and writes only actionable matches (CSP/Diagonal/Vertical/LEAPS) — pure stock-level "overall" verdicts are skipped to keep the dataset clean.
  • capture_backtest(request, response) — persists each backtest verdict with source=backtest.
  • list_recommendations(days, ticker, strategy, closed) / stats(days) — query helpers.

10.2 WeeklyEvaluator

Idempotent runner; skips recs evaluated in the last 6 days. For each open rec:

  1. Fetch current price (yfinance 5d window).
  2. _classify_outcome(action, entry, current, strike, strategy):
    • CSP → winning if current > strike.
    • Sell call → winning if current < strike.
    • Buy strategies → winning if current ≥ entry × 1.02; losing if ≤ entry × 0.98.
  3. Append to outcome_history; increment eval_count (max 4).
  4. After 4 weekly evals (~28 days) → mark closed: true with final_status.

10.3 SignalLearner

Aggregates win/loss/neutral counters from closed recs (180-day window) into signal_stats and verdict_stats Firestore collections. Signals are normalized first (e.g. "RSI 32 — oversold" → "rsi N — oversold") so similar phrasing collapses into a single bucket. A 15-minute in-memory cache fronts the lookup.

Phase 2 (verdict adjustment, currently passive):

  • If (strategy, verdict) win-rate < 30% with n ≥ 10 → downgrade verdict one tier.
  • If win-rate > 70% with n ≥ 20 → upgrade verdict one tier.

10.4 SchedulerManager

  • Cron: Mondays 14:00 UTC (post-market 9 AM ET).
  • Startup catch-up: on every cold start, if meta/eval_status.last_eval_at is older than 6 days, the evaluator runs immediately on a background thread.
  • No-ops cleanly if APScheduler isn’t installed (defensive).

11. End-to-End Request Flows

11.1 Watchlist Scan

  1. User taps Scan in the Compose UI → Retrofit issues GET /api/v1/scan?strong_only=true with X-User-Id.
  2. Backend reads watchlist from users/{uid}/settings/watchlist.
  3. JPFinanceEngine.prefetch_market_data() bulk-downloads 2y OHLCV + fundamentals (8-thread pool).
  4. For each ticker, process_ticker() computes RSI/SMA/BB/ATR/VWAP/MACD/momentum, IV rank, fundamentals, signals → verdict.
  5. Strategy builders fetch option chains from Tradier (fallback yfinance) and emit best 1–2 matches per strategy.
  6. A BackgroundTask calls _tracker.capture_scan_results() → Firestore.
  7. Optionally _filter_strong_buy() trims to STRONG BUY verdicts.
  8. Flat JSON list returned; engine clears its caches; client renders results.

11.2 Async Scan

  1. Client calls /api/v1/scan/async → backend spawns a background job, stores scan_jobs[job_id].
  2. Client polls /api/v1/scan/status/{job_id} every few seconds.
  3. Job progress, then results returned; jobs older than 10 min auto-purge.

11.3 Backtest

  1. Client posts {ticker, strategy, action, strike, premium, expiry} to /api/v1/backtest.
  2. Backend computes price, RSI, SMA, IV, beta, 52w levels; dispatches to e.g. _bt_csp().
  3. get_bt_success() returns historical hit-rate; ROC computed; signals/warnings built; verdict derived.
  4. Result captured into recommendations with source=backtest and returned.

11.4 Weekly Evaluator

  1. Cron triggers WeeklyEvaluator.run() on Monday 14:00 UTC.
  2. Open recs re-priced and classified; outcome history appended.
  3. SignalLearner.update_from_closed_recs(180) recomputes signal_stats / verdict_stats.
  4. Result stored in meta/eval_status; logs counts.

11.5 AI Cross-Validation (client-only)

  1. User opens AI Guru tab on a ticker; AiCrossValidator.validate() runs.
  2. Each configured engine (Claude/Gemini/ChatGPT/Perplexity/Grok) gets the same structured prompt in parallel coroutines.
  3. Engine responses parsed into {verdict, confidence, reasoning}; consensus + agreement % computed.
  4. Result cached in memory 15 min; displayed via AiCrossValidationBadge.

12. Security & Operational Concerns

  • AI keys stored only in EncryptedSharedPreferences (AES256-GCM keys + values).
  • HTTPS enforced (Render-managed TLS).
  • Identity: Google Credential Manager → server trusts X-User-Id for namespacing in Firestore. (Server-side ID-token verification via firebase-admin is available and recommended for production hardening.)
  • Firebase service account JSON ships in the repo for development; should be moved to a Render secret file or GOOGLE_APPLICATION_CREDENTIALS env var, and the JSON rotated.
  • Rate limiting: Tradier capped at ~5 concurrent (≈ 2 req/s within 120 req/min); yfinance paced via Semaphore(1) with 15–20 s back-off on HTTP 429.
  • Single-worker dyno: in-memory scan_jobs table is process-local — horizontally scaling Render to > 1 worker would require moving it to Firestore/Redis.
  • Memory hygiene: per-job cache clearing keeps the dyno bounded across long uptime windows.
  • Notifications: entirely local (WorkManager); no FCM. Easier privacy story but no remote push.
